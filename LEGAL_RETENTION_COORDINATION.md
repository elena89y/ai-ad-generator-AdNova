# 법정 보존기간 정책 반영 — 조율 메모 (한의정 → 김범수)

작성 2026-07-21 · 관련 PR: `feature/legal-retention`

## 배경

고객센터 챗봇/문의 보관 정책을 정하려다, **탈퇴 플로우에 컴플라이언스 버그**를
발견했습니다. `crud/account.py::delete_user_account`가 탈퇴 시 아래 3개 기록을
**하드 삭제**하는데, 전부 전자상거래법상 사업자 보존의무가 걸린 기록입니다.

| 기록 | 종전 동작 | 법정 근거 | 최소 보존 |
|---|---|---|---|
| `SupportInquiry` (1:1 문의) | ❌ 즉시 삭제 | 전자상거래법 시행령 제6조 1항 3호 (소비자 불만·분쟁처리) | **3년** |
| `PurchaseHistory` (구매내역) | ❌ 즉시 삭제 | 동 제6조 1항 2호 (대금결제·재화공급) | **5년** |
| `RefundRequest` (환불신청) | ❌ 즉시 삭제 | 동 (대금결제 관련 분쟁) | **5년** |

→ 탈퇴 회원이 분쟁을 제기하면 사업자가 근거 기록을 못 내는 상태. 시정 필요.

## 조사 결론 (법령 기준)

- **전자상거래법 시행령 제6조**: 위 기간 동안 사업자 보존 **의무**.
- **개인정보보호법 제21조**: 탈퇴 시 개인정보는 원칙적 **지체 없이 파기**하되,
  다른 법령상 보존의무가 있으면 **다른 정보와 분리하여 보관**.

두 법을 합치면 정답은 **"삭제"가 아니라 "가명처리-분리보존 후 기간 경과 시 파기"**.

## 이 PR이 한 것 (한의정)

**설계: "탈퇴회원" 센티넬 계정 방식** (운영 DB가 SQLite라 이 방식을 택함 — 아래 (A) 참조)
탈퇴 시 3종 기록의 `user_id`를 예약된 "탈퇴회원" 센티넬 계정으로 재지정해 개인 식별
링크를 끊는다. `user_id`는 NOT NULL 유지 → FK·기존 관리자 INNER JOIN 전부 무결.

1. **`crud/retention.py` 신설** — 보존기간 상수(3년/5년) + 센티넬 가명처리 + 파기 로직.
2. **모델 3종에 `anonymized_at` 컬럼 추가** (`user_id` 타입 변경 없음 → 마이그레이션 경량).
3. **`delete_user_account` 수정** — 3종은 삭제 대신 `anonymize_legal_records_for_user()`.
   나머지 운영데이터(이미지·광고·히스토리·크레딧·구독·결제수단)는 종전대로 즉시 파기.
4. **`scripts/purge_expired_records.py`** — 보존기간 경과 & **가명처리된** 기록만 파기 배치.
5. **`scripts/migrate_retention.py`** — anonymized_at 컬럼 추가 마이그레이션 (멱등, SQLite 검증).
6. **관리자 회원목록/집계에서 센티넬 제외** (`crud/admin.py`).
7. **개인정보 고지** — 챗봇 KB `faq-acct-003`에 보존기간(문의 3년·결제 5년) 명문화.
8. **테스트** — `test_account.py` 탈퇴 테스트를 가명처리-보존 검증으로 교체 +
   `test_retention.py` 신설(센티넬·활성회원 파기금지·FK 안전성·관리자 조회).

### 적대적 리뷰(5개 렌즈)로 잡아 반영한 것
- **활성 회원 기록 오파기 방지**: purge 에 `anonymized_at IS NOT NULL` 가드 추가
  (안 그러면 활성 유료회원의 5년 지난 결제내역이 배치로 삭제됨).
- **관리자 환불 영구 404 방지**: 센티넬 방식이라 환불/구매 INNER JOIN 이 그대로 동작
  (nullable 방식이었다면 익명화 환불이 관리자 화면에서 사라지고 승인/거절 불가였음).
- **미래 FK 안전**: 탈퇴자가 관리자로 처리했던 환불의 `processed_by_admin_id`도 해제.

## 범수님은 "코드 리뷰 승인"만 하면 됩니다

이 PR의 코드는 전부 작성·검증 완료(478 테스트). 범수님이 다시 코드를 짤 필요 없습니다.
아래 확인 사항도 대부분 **결정/약관**이지 코드 작성이 아닙니다.

### (A) 마이그레이션 — ✅ 자동화됨 (수동 실행 불필요)
`main.py` 가 create_all 직후 `ensure_retention_columns()` 를 호출하도록 넣었습니다 →
앱 기동 시 기존 SQLite DB 에 `anonymized_at` 컬럼이 **자동·멱등 반영**됩니다(레거시 DB
E2E 리허설: 컬럼 추가 + 기존 row 보존 + 재기동 무해 확인). 센티넬 방식이라 테이블
재작성 없이 ADD COLUMN 뿐. **배포 순서 신경 쓸 필요 없음 — 코드만 머지되면 반영.**

### (B) 정책 수치 확정 — 결정(코드 아님)
`crud/retention.py` 상수는 법정 **최소치**(문의 3년 / 결제 5년). 연장 시 이 값만 변경.
기산점: 문의=답변완료일(없으면 접수일), 결제=결제일, 환불=처리일(없으면 신청일).

### (C) 파기 배치 스케줄 (+ 인프라)
`purge_expired_records.py`를 하루 1회 cron/systemd timer 등록 필요(연정님 조율).
미등록 시 조용한 과보관 → **마지막 성공시각 기록·미실행 경보를 배포 체크리스트에** 넣길 권장.

### (D) 잔여 정책 결정 (문서화 필요)
- **"가명처리" ≠ 완전 익명화**: 본문(문의 content, 환불 reason, 결제 description/식별자)은
  분쟁 근거라 보존하므로 재식별 여지가 남습니다. 본문 마스킹 여부는 **법무 판단** 권장.
- **미처리(pending) 건 파기 경계**: 아직 답변/처리 안 된 오래된 건은 접수일 기준으로
  파기될 수 있음 — 진행 중 분쟁 보호가 필요하면 "미해결 건은 파기 제외" 규칙 추가 검토.
- **운영/품질 로그**(서버로그·Langfuse 질문 텍스트)는 분쟁기록과 별개 트랙 —
  보관기간(예: 90일) 명시 권장. 정책 결정 사항.

### (E) 개인정보처리방침·이용약관 본문
KB(`faq-acct-003`)에 "문의 3년·결제 5년 분리보관" 문구를 넣었지만 **정본은
개인정보처리방침**입니다. 방침 본문에도 동일 문구 반영 필요(법무 검토 후).
